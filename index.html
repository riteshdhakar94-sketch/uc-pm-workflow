End-to-End Workflow
5 Phases. 4 Skills. One Loop.
Each phase has a defined input, a skill that processes it, and a specific output that feeds the next phase.
1
Intake
Validate the Requirement
Before anything is prioritised or built — is this a real, well-defined problem?
🔍
cr-review
Trigger: paste a Jira CR from ops / sales / CS and say "review this CR"
New Skill ▼
📥 Input
Rough Jira CR ticket from ops / sales / CS
📤 Output
Structured CR: Use Case → Problem → Impact → Expected Outcome
Section scores: ✅ Clear / ⚠️ Weak / ❌ Missing
Up to 5 targeted clarification questions
Verdict: Accept / Needs Clarification / Duplicate / Reject
Key Rules
✅ Accept → moves to Discovery (PRD)
⚠️ Needs Clarification → send back to requester with specific questions (not generic ones)
❌ Reject → respond with reason: config issue / bug / out of scope / support ticket
🔁 Possible Duplicate → check against backlog before accepting

3
Build & Handoff
Write the Ticket. Review It. Hand It Off.
Convert the approved PRD into an engineering-ready Jira ticket, then score it for completeness before handing to the team.
🎫
jira-ticket
Trigger: "make this a ticket", "turn this into a story", "write a Jira ticket for X"
Updated ▼
📥 Input
Approved PRD from Phase 2
Problem description (informal is fine)
Support escalation or API issue description
📤 Output
Background → Pain Point → Solution → Scope of Work → Testing → Release Notes
User story with correct persona
Option tables, fallback behaviour, QA checklist
Built-in Checks After Every Ticket
🎨 Design handoff: scans ticket for UI signals (new page, modal, form, table column, badge, button). If found → prompts to build a mock using the unicommerce-design-system skill. Silent if backend-only.
📣 Release comms prompt: always closes with — "When this ships, paste it back and tell me the work type → I'll generate the seller announcement, internal team update, changelog entry, and Notion release page."
✅
prd-jira-review
Trigger: "review this ticket", "is this ready to hand off?", "check this before I share it"
New Skill ▼
📥 Input
Jira ticket (any format, pasted text)
Strategy PRD → auto-detected and redirected
📤 Output
Section scorecard: ✅/⚠️/❌ per section
Domain checks: API impact, seller impact, feasibility
Verdict + top 3 fixes before handoff
Smart Detection (Step 0)
🔎 Detects if input is an engineering ticket (Background/Release Notes format) → runs full scorecard
📄 Detects if input is a strategy PRD (Success Criteria/Sign-off format) → stops and redirects to jira-ticket skill first
⚠️ If ambiguous → scores what it can, flags the mismatch explicitly
4
Ship
Generate Release Communications
When the feature ships, paste the ticket back with the work type — get all 4 comms outputs in one go.
📣
jira-ticket · Release Comms Mode
Trigger: "this shipped", "generate release comms", "write the announcement", "draft the seller email"
▼
Work type → tone
FeatureBenefit-led · "You can now…"
EnhancementPractical · "We've improved…"
Bug FixReassuring · "We've resolved…"
Integration UpdateTechnical-but-clear
📧 Seller EmailSubject + what changed + how to use + opt-in or automatic
💬 Internal Slack UpdateWhat / Why / Impact / Status / Owner — in Slack format
📋 Changelog EntryOne line. Present tense. No jargon. Month Year format.
📄 Notion Release PageSummary / What changed / Seller impact / Ops impact / Known limits
🚫 Never includes in seller comms: method names, script names, API fields, internal ticket IDs, or the word "bug" in feature/enhancement announcements
5
Measure
Did It Work?
4–8 weeks after launch, run a post-launch review. Score each success criterion. Feed findings back into the intake queue.
📊
prd-generator · Mode B
Trigger: "post-launch review", "how did X perform", "did we hit our goals", "check metrics against this PRD"
▼
📥 Input
Original PRD (with Success Criteria)
Post-launch metric data (paste or describe)
Time since launch
📤 Output
Metrics scorecard: ✅ Hit / ⚠️ Partial / ❌ Missed / ❓ No Data
Root cause analysis for misses
Next actions: Continue / Investigate / Fix / Measure / Rethink
Verdict → feeds back to Phase 1 intake
Important Nuances
⏱️ If < 4 weeks since launch → lagging indicators are too early. Focus on leading indicators only.
🔍 Distinguishes "feature works but adoption is low" vs "feature doesn't work" — different fixes
📈 Flags uncertain attribution — was it the feature, seasonality, a campaign, or a platform change?

Reference
Canonical Unicommerce Ticket Format
Every Jira ticket — regardless of feature area or integration — must follow this section order. These are the rules the jira-ticket and prd-jira-review skills enforce.
Section What it must contain Common mistake
📘 Background Current system behaviour only. Method names, API fields, formulas. What happens today before any change. Describing the proposed fix here instead of current state
🔴 Pain Point Root cause in one line, then numbered operational consequences: pickup refusal, billing mismatch, manual reconciliation, seller escalation. "Causes issues" — not a pain point. Name the consequence.
✅ Solution One paragraph. Name any new UI element explicitly. Option table if configurable. Default/fallback behaviour always stated. Multiple problems in one section. UI element unnamed. Blank config not addressed.
🔧 Scope of Work User story with correct persona. Numbered tasks specific enough to assign. In Scope / Out of Scope drawn explicitly. Related flows covered. Persona too generic ("user" or "admin"). Tasks vague. In/Out of scope missing.
🧪 Testing Table with: happy path, edge cases, blank/default config (backward compat), reverse flow if applicable. Expected results specific. Happy path only. No backward compat test. Expected results vague ("should work").
📣 Release Notes Written for sellers/ops. What changed, why it matters, how to use it. Opt-in vs automatic. No method names or jargon. Under 10 lines. Technical jargon. Doesn't state if opt-in or automatic. No "Applicable to" line.
Team Rules
How We Use This System
Non-negotiables for the PM team and engineering team.
📋

PM Rules

For Ritesh and the product team
1
No CR moves to the backlog without passing cr-review. Slack messages and verbal asks are not accepted requirements.
2
No PRD goes to engineering without a sign-off table. Product, Engineering, and Design must all be checked.
3
Every Jira ticket is reviewed with prd-jira-review before it enters a sprint. Verdict must be "Ready to hand off" or "Needs minor fixes."
4
If a ticket has UI changes, a mock must accompany it. Use the unicommerce-design-system skill — the jira-ticket skill will prompt you.
5
Every shipped feature gets a post-launch review at 4 and 8 weeks. Use prd-generator Mode B. Findings go back into the intake queue.
⚙️

Engineering Rules

What to expect from every ticket
1
Every ticket will have all 6 sections. If a section is missing, return it to PM before starting work — not mid-sprint.
2
Background = current state only. If the Background section describes a solution, flag it — it's a red flag for scope clarity.
3
Default/fallback behaviour will always be stated. If it's not — ask. Never assume.
4
UI tickets will include a mock in Unicommerce design language. Don't start UI work without it.
5
When a ticket ships, tell the PM the work type (Feature / Enhancement / Bug Fix / Integration update) — this triggers the release comms workflow.
